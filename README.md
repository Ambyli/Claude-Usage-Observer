# Claude Usage Widget

A Windows system tray widget that displays your Claude Code token usage in real time, with rolling average limits and a live refresh countdown.

---

## Requirements

```bash
pip install pystray Pillow
```

## Running

```bash
python claude_usage_widget.py
```

The widget appears in your system tray. Hover to see a quick summary; click to open the full popup.

---

## How data is collected

Claude Code (CLI and desktop app) writes a `.jsonl` file for every session under:

```
~/.claude/projects/<project-name>/<session-id>.jsonl
```

Each line is a JSON object representing one event in the conversation. Completed assistant turns include a `message.usage` object like:

```json
{
  "message": {
    "usage": {
      "input_tokens": 3,
      "cache_creation_input_tokens": 5850,
      "cache_read_input_tokens": 11185,
      "output_tokens": 512
    }
  },
  "timestamp": "2026-04-13T19:04:48.000Z"
}
```

The widget scans all `.jsonl` files across all projects in a single pass, keeping only lines where `output_tokens > 0` (i.e. completed turns, not streaming-in-progress events). Tokens are bucketed by the UTC date of their `timestamp`.

### Token fields summed per turn

| Field | What it counts |
|---|---|
| `input_tokens` | Uncached prompt tokens sent to the model |
| `cache_creation_input_tokens` | Tokens written into the prompt cache |
| `cache_read_input_tokens` | Tokens read back from the prompt cache |
| `output_tokens` | Tokens generated by the model |

All four are added together to form the **total** for a turn. Input and output are tracked separately for the breakdown display.

---

## How usage is calculated

### Daily total

All tokens from completed assistant turns where the timestamp falls on today's UTC date.

### Weekly total

All tokens from the current **calendar week** — Monday through today (inclusive).

---

## How limits are calculated

Limits are computed automatically from your own history on every refresh — no manual configuration required.

### Daily limit

> Average daily token usage over the **7 days before today**, excluding configured weekdays and zero-usage days.

```
daily_limit = mean( tokens(d) for d in [yesterday … yesterday-6]
                    if d.weekday() not in EXCLUDE_WEEKDAYS
                    and tokens(d) > 0 )
```

### Weekly limit

> Average total token usage across the **7 previous complete calendar weeks**, with excluded weekdays omitted from each week's sum.

```
weekly_limit = mean( week_total(w) for w in [last_week … 7_weeks_ago]
                     where week_total sums only days not in EXCLUDE_WEEKDAYS
                     and week_total > 0 )
```

Weeks with zero total (e.g. holiday weeks) are excluded before averaging. The widget looks back up to 63 days to cover all 7 prior weeks.

---

## Progress bars

Each bar compares current usage against the computed rolling average limit.

| Colour | Meaning |
|---|---|
| Green | Below 70% of average |
| Amber | 70–89% of average |
| Red | 90%+ of average |

The label underneath shows: `X.X%  (Y remaining  (avg Z/day))`

Going over 100% is possible on heavy days — the bar simply fills completely and the percentage exceeds 100%.

---

## Refresh behaviour

- Data is re-read from disk every **5 minutes** (configurable via `REFRESH_INTERVAL_SECONDS`).
- The popup shows a live countdown to the next refresh, ticking every second.
- "Refresh Now" in the right-click tray menu forces an immediate re-scan.
- The tray icon dot turns **yellow** while refreshing and **red** on error.

## Configuration

Settings live in `.env` in the same directory as the script:

```env
# Comma-separated path prefixes. Only sessions whose working directory starts
# with one of these will be counted. Leave blank to include all sessions.
INCLUDE_PATHS=C:\Users\Amber Price\OneDrive\Documents\Zeo
```

### Account filtering

Claude Code does not record which Anthropic account generated a session in its local log files. The only distinguishing information available is the working directory (`cwd`) recorded on each log line.

**How project folders are named**

When Claude Code starts a session in a directory, it derives a folder name from that path and stores the session logs there:

```
~/.claude/projects/C--Users-Amber-Price-OneDrive-Documents-Zeo-phoenix/
~/.claude/projects/C--Users-Amber-Price-Desktop-claude-usage/
```

Path separators and colons become dashes. All sessions for the same working directory share the same project folder, regardless of which account was logged in at the time.

**Filtering by path**

If you use multiple accounts across different project directories, set `INCLUDE_PATHS` to the root path(s) belonging to the account you want to track:

```env
# Work account only
INCLUDE_PATHS=C:\Users\Amber Price\OneDrive\Documents\Zeo

# Multiple roots (comma-separated)
INCLUDE_PATHS=C:\Users\Amber Price\OneDrive\Documents\Zeo,C:\Users\Amber Price\Desktop\claude usage
```

Matching is **case-insensitive** and **prefix-based** — a single root path automatically covers all subdirectories beneath it. Leave `INCLUDE_PATHS` blank to count all sessions regardless of location.

**Limitation**

If both accounts have been used in the same working directory, their sessions will be mixed in the same project folder and cannot be separated. The only workaround is to keep each account's work in distinct directory trees.

`EXCLUDE_WEEKDAYS` — comma-separated day numbers to skip when computing averages (0=Monday, 6=Sunday). Defaults to `5,6` (Saturday and Sunday). Days are still counted in the displayed totals; they are only excluded from the limit baseline.

```env
# Exclude weekends (default)
EXCLUDE_WEEKDAYS=5,6

# Exclude Mon and Fri as well (e.g. short work week)
EXCLUDE_WEEKDAYS=0,4,5,6
```

No API key is required. All data is read locally.
